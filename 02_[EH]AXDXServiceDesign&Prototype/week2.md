## 📚 주요 내용

### 1. 프로젝트 캔버스 개요

#### 1.1 정의 및 목적

**프로젝트 캔버스 (Project Canvas)**:
- 프로젝트의 핵심 요소를 한눈에 시각화한 전략 도구
- 팀원 간 공통 이해 형성 및 의사소통 촉진
- 초기 기획 단계에서 빠른 검증 및 수정 가능

**목적**:
- 프로젝트 방향성 명확화
- 이해관계자 정렬 (Alignment)
- 핵심 가정 및 위험 요소 파악
- 빠른 의사결정 지원

#### 1.2 BMC vs Project Canvas

| 항목 | BMC (Business Model Canvas) | Project Canvas |
|------|----------------------------|----------------|
| **초점** | 비즈니스 모델 | 프로젝트 실행 |
| **대상** | 스타트업, 신사업 | 특정 프로젝트 |
| **시간 범위** | 중장기 (1-3년) | 단기 (3-6개월) |
| **핵심 질문** | "어떻게 돈을 벌 것인가?" | "무엇을 언제까지 만들 것인가?" |

### 2. 프로젝트 캔버스 9가지 구성요소

#### 2.1 Goal (목표)

**질문**: "무엇을 달성하려고 하는가?"

**내용**:
- 명확하고 측정 가능한 목표
- SMART 기준 적용 (Specific, Measurable, Achievable, Relevant, Time-bound)

**예시**:
- ❌ 나쁜 목표: "고객 만족도를 높인다"
- ✅ 좋은 목표: "6개월 내 NPS 점수를 40에서 60으로 향상시킨다"

#### 2.2 Problem (문제)

**질문**: "어떤 문제를 해결하려고 하는가?"

**내용**:
- 타겟 사용자가 겪는 구체적인 Pain Point
- 정량적 데이터로 뒷받침 (시장 규모, 빈도, 영향)

**예시**:
- ❌ 나쁜 문제 정의: "사람들이 쇼핑을 어려워한다"
- ✅ 좋은 문제 정의: "20-30대 직장인들이 퇴근 후 쇼핑 시간이 부족하여 (평균 주 1회), 필요한 물건을 제때 구매하지 못해 불편함을 느낀다 (설문 응답자 78%)"

#### 2.3 Solution (솔루션)

**질문**: "어떻게 문제를 해결할 것인가?"

**내용**:
- 핵심 기능 또는 서비스 (3-5개)
- 기존 솔루션 대비 차별점

**예시**:
```
문제: 직장인 쇼핑 시간 부족
솔루션:
1. 24시간 온라인 주문 가능
2. 새벽 배송 (주문 후 12시간 이내)
3. AI 추천 알고리즘 (구매 이력 기반)
```

#### 2.4 Value Proposition (가치 제안)

**질문**: "왜 우리 솔루션을 선택해야 하는가?"

**내용**:
- 사용자가 얻는 핵심 가치
- 경쟁사 대비 우위

**예시**:
- "직장인이 퇴근 후 주문하면, 다음 날 아침 출근 전에 받을 수 있는 유일한 서비스"

#### 2.5 Metrics (지표)

**질문**: "성공을 어떻게 측정할 것인가?"

**내용**:
- KPI (Key Performance Indicator)
- 목표 수치 및 달성 시점

**예시**:
| 지표 | 현재 | 목표 (3개월 후) | 측정 방법 |
|------|------|----------------|-----------|
| 월간 활성 사용자 (MAU) | 0 | 10,000 | 앱 로그 분석 |
| 구매 전환율 (CVR) | - | 15% | 주문 수 / 방문자 수 |
| 고객 만족도 (NPS) | - | 50 이상 | 설문 조사 |

#### 2.6 Stakeholders (이해관계자)

**질문**: "누가 이 프로젝트에 영향을 받거나 영향을 주는가?"

**내용**:
- Primary: 직접적 영향 (사용자, 고객)
- Secondary: 간접적 영향 (파트너, 투자자)
- Tertiary: 관찰자 (규제기관, 언론)

**예시**:
- **Primary**: 20-30대 직장인 (타겟 고객), 물류 파트너
- **Secondary**: 투자자, 광고주
- **Tertiary**: 전자상거래 규제기관, 소비자 보호단체

#### 2.7 Team (팀)

**질문**: "누가 무엇을 담당하는가?"

**내용**:
- 팀원 역할 및 책임
- RACI 매트릭스 참조

**예시**:
| 이름 | 역할 | 책임 |
|------|------|------|
| 김민수 | PM | 프로젝트 관리, 일정 조율 |
| 이지은 | UX/UI Designer | 와이어프레임, 시각 디자인 |
| 박준호 | Frontend Dev | React 앱 개발 |
| 최수진 | Backend Dev | API 개발, DB 설계 |

#### 2.8 Budget (예산)

**질문**: "얼마의 비용이 필요한가?"

**내용**:
- 인건비, 도구/인프라 비용, 마케팅 비용
- 자금 조달 계획

**예시**:
| 항목 | 금액 | 비고 |
|------|------|------|
| 인건비 | ₩40,000,000 | 4명 × 3개월 |
| 클라우드 인프라 | ₩3,000,000 | AWS, DB |
| 디자인 도구 | ₩500,000 | Figma, Adobe CC |
| 마케팅 | ₩10,000,000 | SNS 광고, 인플루언서 |
| **합계** | **₩53,500,000** | |

#### 2.9 Risks (위험 요소)

**질문**: "무엇이 잘못될 수 있는가?"

**내용**:
- 기술적, 시장, 운영 위험
- 완화 전략

**예시**:
| 위험 | 영향 | 확률 | 완화 전략 |
|------|------|------|-----------|
| 물류 파트너 계약 실패 | 높음 | 중간 | 복수 파트너 확보, 자체 배송 검토 |
| 경쟁사 유사 서비스 출시 | 중간 | 높음 | 빠른 MVP 출시, 차별화 기능 강화 |
| 개발 일정 지연 | 높음 | 중간 | 애자일 스프린트, 주간 진행 체크 |

### 3. 좋은 캔버스 vs 나쁜 캔버스

#### 3.1 좋은 캔버스의 특징

**명확성 (Clarity)**:
- 누구나 이해 가능한 간결한 문장
- 전문 용어 최소화 또는 설명 추가

**구체성 (Specificity)**:
- 숫자와 데이터로 뒷받침
- "많이", "빠르게" 같은 모호한 표현 지양

**현실성 (Feasibility)**:
- 리소스 및 시간 제약 고려
- 과도한 목표 설정 지양

**측정 가능성 (Measurability)**:
- 명확한 KPI 및 목표 수치
- 측정 방법 명시

#### 3.2 나쁜 캔버스의 특징

**모호함**:
- "혁신적인 서비스로 고객 만족도를 높인다" (어떻게? 얼마나?)

**비현실적 목표**:
- "3개월 내 100만 사용자 확보" (시드 투자 없이, 팀 3명)

**측정 불가능한 지표**:
- "브랜드 인지도 향상" (측정 방법 불명확)

**위험 요소 무시**:
- 경쟁사, 기술적 난이도, 규제 리스크 고려 없음

#### 3.3 Before & After 예시

**Before (나쁜 캔버스)**:
```
Goal: 멋진 앱을 만든다
Problem: 사람들이 불편해한다
Solution: 새로운 기능을 추가한다
Value Proposition: 최고의 서비스
Metrics: 사용자가 늘어난다
```

**After (좋은 캔버스)**:
```
Goal: 6개월 내 월간 활성 사용자 10,000명 달성
Problem: 20-30대 직장인 78%가 퇴근 후 쇼핑 시간 부족 (설문, n=500)
Solution: 24시간 주문 + 새벽 배송 + AI 추천
Value Proposition: 유일한 새벽 배송 서비스, 주문 후 12시간 이내 도착
Metrics: MAU 10,000, CVR 15%, NPS 50+
```

### 4. SMART 목표 설정

#### 4.1 SMART 프레임워크

**S - Specific (구체적)**:
- 명확하고 구체적인 목표
- 5W1H 활용 (Who, What, When, Where, Why, How)

**M - Measurable (측정 가능)**:
- 숫자로 표현 가능
- 진행 상황 추적 가능

**A - Achievable (달성 가능)**:
- 현실적이고 실행 가능
- 리소스 및 역량 고려

**R - Relevant (관련성)**:
- 상위 목표 또는 비전과 연결
- 비즈니스 가치 제공

**T - Time-bound (기한 설정)**:
- 명확한 마감일
- 중간 마일스톤 포함

#### 4.2 SMART 목표 예시

**예시 1: 사용자 확보**
- ❌ 비SMART: "사용자를 늘린다"
- ✅ SMART: "3개월 내 월간 활성 사용자(MAU)를 0명에서 5,000명으로 증가시킨다. (마케팅 예산 1,000만원, SNS 광고 및 인플루언서 협업)"

**예시 2: 수익 창출**
- ❌ 비SMART: "돈을 많이 번다"
- ✅ SMART: "6개월 내 월 매출 5,000만원 달성 (평균 객단가 5만원 × 월 1,000건 주문)"

**예시 3: 고객 만족도**
- ❌ 비SMART: "고객 만족도를 높인다"
- ✅ SMART: "3개월 내 NPS(Net Promoter Score)를 40에서 60으로 향상시킨다. (고객 설문 월 1회, 최소 응답자 100명)"

#### 4.3 목표 분해 (Goal Decomposition)

**Big Goal → Small Goals**

**예시: MAU 10,000명 달성**

1. **월별 목표 설정**:
   - 1개월차: MAU 1,000
   - 2개월차: MAU 3,000
   - 3개월차: MAU 10,000

2. **주간 신규 사용자 목표**:
   - 1개월차: 주 250명 (1,000 / 4주)
   - 2개월차: 주 500명
   - 3개월차: 주 1,750명

3. **일일 활동 목표**:
   - SNS 포스팅: 일 1회
   - 인플루언서 협업: 주 2회
   - 광고 클릭: 일 100회 (CTR 2%, CVR 10% → 일 10명 가입)

### 5. AI 챗봇 서비스 기획 사례

#### 5.1 프로젝트 개요

**서비스명**: "SmartBot AI 고객센터"

**배경**:
- 기존 고객센터 상담원 부족 (월 평균 5,000건 문의, 응답률 60%)
- 고객 대기 시간 평균 15분
- 단순 반복 질문 70% (배송 조회, 반품 절차 등)

#### 5.2 완전한 프로젝트 캔버스

```
+----------------------------------------------------------+
|  Goal              |  Problem          |  Solution      |
|                    |                   |                |
| 3개월 내 고객 문의  | 고객센터 상담원   | 1. AI 챗봇     |
| 자동 응답률 80%    | 부족으로 인한     | 24/7 운영      |
| 달성, 평균 응답    | 긴 대기시간       | 2. FAQ 자동응답|
| 시간 1분 이내      | (평균 15분)       | 3. 에스컬레이션|
| 단축               |                   | 4. 주문조회    |
|                    |                   | 자동화         |
+----------------------------------------------------------+
|  Value Prop        |  Metrics          |  Stakeholders  |
|                    |                   |                |
| • 고객: 즉시 응답, | • 자동응답률:     | Primary:       |
| 24시간 가능        | 0% → 80%          | • 고객         |
| • 회사: 상담 비용  | • 응답시간:       | • 상담원       |
| 40% 절감, 고객     | 15분 → 1분        | Secondary:     |
| 만족도 향상        | • CSAT:           | • IT팀, 경영진 |
|                    | 3.2 → 4.5/5.0     |                |
+----------------------------------------------------------+
|  Team              |  Budget           |  Risks         |
|                    |                   |                |
| • PM: 김프로       | • 인건비:         | • AI 오답률    |
| • AI Eng: 이에이아이| ₩30M (4명×3월)   | 높음 → 학습데이|
| • Backend: 박백엔드 | • GPT-4 API:     | 터 1,000건확보 |
| • UX: 최유엑스      | ₩3M              | • 개인정보유출 |
|                    | • 인프라: ₩2M     | → 데이터암호화 |
|                    | 합계: ₩35M        |                |
+----------------------------------------------------------+
```

#### 5.3 상세 설명

**Goal (SMART 적용)**:
- **S**: 고객 문의 자동 응답률 80% 달성
- **M**: 응답률 (%), 응답 시간 (분), CSAT 점수
- **A**: GPT-4 기술 활용, 3개월 개발 기간
- **R**: 고객 서비스 개선 → 매출 증대 연결
- **T**: 3개월 (명확한 마감일)

**Problem (정량적 데이터)**:
- 월 평균 5,000건 문의 → 응답률 60% (2,000건 미응답)
- 평균 대기 시간 15분 → 고객 불만족 증가
- 단순 질문 70% → 상담원 리소스 낭비

**Solution (구체적 기능)**:
1. GPT-4 기반 자연어 처리
2. 기존 FAQ 데이터베이스 연동
3. 복잡한 문의는 상담원 전환
4. 실시간 주문/배송 조회 API 연동

**Metrics (추적 가능한 지표)**:
- 기술적: 응답 정확도, API 응답 시간
- 비즈니스: CVR, 고객 만족도
- 운영적: 상담 비용, 처리량

### 6. 프로젝트 캔버스 작성 실습

#### 6.1 작성 순서 (권장)

1. **Problem (문제)**: 무엇을 해결하려고 하는가?
2. **Solution (솔루션)**: 어떻게 해결할 것인가?
3. **Value Proposition (가치 제안)**: 왜 우리 솔루션인가?
4. **Goal (목표)**: SMART 목표 설정
5. **Metrics (지표)**: 어떻게 측정할 것인가?
6. **Stakeholders (이해관계자)**: 누가 관련되어 있는가?
7. **Team (팀)**: 누가 무엇을 하는가?
8. **Budget (예산)**: 얼마나 필요한가?
9. **Risks (위험)**: 무엇이 잘못될 수 있는가?

#### 6.2 작성 팁

**Tip 1: 사용자 인터뷰 먼저**
- 가정이 아닌 실제 데이터 기반 작성
- 최소 5명 이상 인터뷰

**Tip 2: 한 페이지에 요약**
- A4 1장 또는 캔버스 템플릿 활용
- 간결하고 명확하게

**Tip 3: 팀과 함께 작성**
- 워크숍 형식 (1-2시간)
- 포스트잇 활용, 브레인스토밍

**Tip 4: 주기적 업데이트**
- 1-2주마다 재검토
- 가정 검증 후 수정

5. 최종 버전으로 정리

### 8. 캔버스 작성 워크숍

#### 8.1 워크숍 준비

**사전 준비**:
- 참여자: 팀원 3-7명
- 시간: 90-120분
- 장소: 회의실 또는 온라인
- 도구: 포스트잇, 마커, 캔버스 템플릿

**준비물**:
- 사용자 인터뷰 결과
- 경쟁 분석 자료
- 시장 조사 데이터
- 기술 검토 결과

#### 8.2 워크숍 진행

**Phase 1: 개별 브레인스토밍 (20분)**
- 각자 포스트잇에 아이디어 작성
- 하나의 포스트잇에 하나의 아이디어

**Phase 2: 공유 및 그룹핑 (30분)**
- 모든 아이디어 벽에 붙이기
- 유사한 아이디어 그룹핑
- 중복 제거

**Phase 3: 우선순위 및 정제 (40분)**
- 각 블록별로 최고 아이디어 선택
- 구체적으로 정제
- 데이터 및 근거 추가

**Phase 4: 검토 및 피드백 (30분)**
- 전체 캔버스 일관성 검토
- 빠진 부분 확인
- 다음 액션 아이템 정의

#### 8.3 원격 워크숍 팁

**도구 활용**:
- Zoom/Meet: 화상 회의
- Miro: 실시간 협업 보드
- Slack: 채팅 및 파일 공유

**진행 방법**:
- 화면 공유 + 음성 설명
- 브레이크아웃 룸 활용 (소그룹 토론)
- 투표 기능으로 우선순위 결정

### 9. 캔버스 검증 및 개선

#### 9.1 검증 방법

**내부 검증**:
- 팀원 리뷰 (Cross-check)
- 멘토/어드바이저 피드백
- 기존 성공 사례와 비교

**외부 검증**:
- 잠재 고객 인터뷰 (5-10명)
- 전문가 자문 (산업 전문가)
- 파일럿 테스트 (작은 규모)

#### 9.2 일반적인 문제점과 해결책

**문제 1: 목표가 너무 추상적**
- 해결책: 구체적 숫자로 표현, 측정 방법 명시

**문제 2: Problem-Solution 미스매치**
- 해결책: 사용자 인터뷰 재실시, Pain Point 재확인

**문제 3: 비현실적인 예산/일정**
- 해결책: 유사 프로젝트 벤치마킹, 전문가 자문

**문제 4: 팀 역할 모호**
- 해결책: RACI 매트릭스 상세 작성

#### 9.3 지속적 개선

**주기적 업데이트**:
- 주 1회: 진행 상황 체크
- 월 1회: 캔버스 전체 리뷰
- 분기 1회: 목표 및 전략 조정

**학습 반영**:
- 사용자 피드백 수집
- 경쟁사 동향 파악
- 기술 트렌드 반영
